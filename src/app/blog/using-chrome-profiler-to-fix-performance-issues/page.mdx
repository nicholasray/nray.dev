When you go to a doctor for pain, they might take an x-ray, CT scan, or MRI of
you to help with the diagnosis. Doctors then look at the results of the imaging
and map certain characteristics to a diagnosis. Similar to how the medical world
has diagnostic tests for humans, developers have tests that can help them
determine issues with websites. One of the most popular ones, Chrome DevTools,
contains a plethora of helpful tools to help diagnose issues with the
performance of a website.

In this post, we'll use the performance panel of Chrome DevTools to diagnose
several common performance issues that can plague websites. In doing so, we'll
learn the characteristics of each problem so that you can recognize them in the
future. Let's get started!

## How to Profile a Website with Chrome Developer Tools

Using Chrome DevTools to profile a website can be done by performing the
following steps:

1. Open a new window of Chrome in
   [incognito mode](https://support.google.com/chrome/answer/95464). This helps
   reduce unnecessary noise in your performance profiles caused by factors such
   as your browser's installed extensions which you probably don't care about.
   Incognito Mode disables extensions by default so they won't influence your
   profiling results.
2. Go to the website you are interested in profiling.
3. Open DevTools with Command+Option+I (Mac) or Control+Shift+I (Windows).
4. Click the "Performance" tabs in DevTools.

<Image src={performancePanel} alt="Chrome DevTools performance panel" />

5. Make sure the "Screenshots" and "Enable advanced paint instrumentation
   (slow)" checkboxes are checked to get additional helpful data from the
   profile. Additionally, in the likely event that the website you are profiling
   serves users on mobile devices or low-end devices, change "CPU" to "4x
   slowdown" or "6x slowdown" to get a more realistic picture of how the website
   will perform on these devices.
6. Click the "Record" button to start profiling the runtime performance of the
   website.

<Image
  src={performancePanelRecord}
  alt="Chrome DevTools performance panel with the record button highlighted."
/>

7. As it's recording, perform any action that you want to measure such as
   scrolling or clicking a button.
8. After you have performed the action, click the "Stop" button.

<Image
  src={performancePanelStop}
  alt="Chrome DevTools performance panel with the stop button highlighted."
/>

9. Chrome displays the results of your profile. This amount of data might be a
   bit intimidating, but we'll cover a lot of in the subsequent sections!

<Image
  src={performancePanelResults}
  alt="Chrome DevTools performance panel with profiling results shown."
/>

The previous steps show how you profile certain **actions** on a website like
clicking a button or scrolling, but what if you just wanted to measure the
loading performance of a website? Well, you can do that too! In fact, this is
such a frequent use case that the Chrome team added a button to the Performance
panel that lets you do this. Instead of clicking the "Record" button mentioned
in the previous steps, you would click the rounded arrow button next to it.

<Image
  src={performancePanelRecordPageLoad}
  alt="Chrome DevTools performance panel with the button to profile page load highlighted."
/>

This initiates a reload of the page and starts the profiler. After the page
loads, the profiler will automatically stop and display the results.

## Animations That Run On the Main Thread

You might want to use animations on your website to improve the user experience.
However, because the animation of certain CSS properties are prone to sluggish
performance, it may actually have the opposite effect.

It turns out that limiting your animations to the `opacity` and/or `transform`
CSS properties will give you the best chance at achieving performant animations.
Why? Animating any other CSS property will cause it to run on the browser's
[main thread](https://developer.mozilla.org/en-US/docs/Glossary/Main_thread)
which can easily become bogged down with other work.

However, animating `opacity` and `transform` properties leverages another
browser thread called the
[compositor thread](https://web.dev/animations-overview/). This thread typically
has a lot less competition than the main thread. Less competition means that
your animations have a much higher chance of looking smooth and performant to
the user.

<JankAnimation duration={2} />

In the example above, there are two animations. The one on the left animates the
ball's `top` CSS property while the one on the right animates the ball's
`transform` property. As a result, the animation on the left is running on the
main thread, while the animation to the right is leveraging the compositor
thread.

Now click the button below these animations to observe what happens when the
main thread is busy for 2 seconds. You should notice that the animation on the
main thread grinds to a halt while the compositor thread animation keeps running
smoothly.

Now let's profile what is happening. Follow the
[steps above](#how-to-profile-a-website-with-chrome-developer-tools) and click
the button as part of step 7. After you've clicked the stopped button, a profile
that is similar to the one below should load:

As shown, in the image above, you'll notice a long yellow bar labeled `click`
under the expanded "Main" section. This section shows all activity that occurred
on the main thread during the recording and the bar corresponds to your button
click. Hovering over this yellow bar displays a tooltip that indicates it took 2
seconds. You'll also notice that this bar is stacked on top of other bars.
Chrome DevTools shows main thread activity via a flame chart so the y-axis
represents the call stack while the x-axis represents time. Therefore, this
stack of bars represents the call stack that occurred when you clicked the
button.

A common culprit of sluggish performance is too much activity on the main
thread. According to the [RAIL](https://web.dev/rail/#focus-on-the-user) model,
animations should produce a frame every 16ms in order to look smooth to the
user. However, if the animation occurs on the main thread, long-running
JavaScript tasks, such as the long click handler in the demo, can block the
animation from producing any new frames until the JS has finished. That's why
its critical for the animations to run on the compositor thread and for us to be
careful how long our JS runs on the main thread.

## Forced Synchronous Layouts

When the browser produces a frame that the user can see, the following steps can
occur:

<Image
  src={renderingSteps}
  alt="The browser's rendering steps. Each step is represented by a box."
/>

1. JavaScript executes
2. Style calculations. Browser determines the CSS rules and elements that are
   involved in the changes
3. Layout. Browser determines the position and size of elements
4. Paint. Browser records draw calls
5. Composite. Browser calculates layer info and passes it to compositor thread

Ideally, the browser won't need to repeat steps when producing a frame. However,
what happens if JavaScript mutates the DOM (step 1) and then queries layout
information about the element it mutated (step 3)?

```js
const box = document.querySelector(".box");
// Write
box.classList.add("box--large");
// Read
const width = box.getBoundingClientRect().width;
```

In the example above, we add a class to an element and then query its width.
However, what if the newly added class affects the width? In order to calculate
this info, the browser is **forced** to run steps 2 and 3 earlier than it would
have otherwise. In fact, it runs both of these steps **while** the JS is
executing.

This is called a `forced synchronous layout` and it can become particularly
expensive when a write follows a read which follows a write as it forces the
browser to perform multiple calculations of layout instead of one before the
frame is produced. How do we prevent this inefficiency? It turns out that the
ordering of these calls matters a lot.

<div>
  <a href={demoUrl}>Forced Synchronous Layout Demo</a>
</div>

Let's see an example of this. Visit the above link and you'll notice a box at
the bottom with its height and left offset position. At the top, there are two
strategies of changing the box's height.

The strategy on the left, "Unbatched reads/writes", iterates over each box
(currently only one), reads the current height of the box, increments the height
of each box by 10 pixels and writes it in its `innerText`. After that, it
performs another read/write set by calculating the relative left position of the
box and writing that in its `innerText`. Importantly, there is a **read**,
followed by a **write**, followed by a **read**, followed by a **write**.

The strategy on the right, "Batched reads/writes", similarly iterates over each
box, but batches the two reads of `height` and `offsetLeft` together before
performing the writes. Furthermore, instead of performing the writes in the same
frame, it defers the execution of the writes in a callback queue and waits until
another frame passes before executing the callbacks. This is accomplished by
wrapping the writes in two nested `requestAnimationFrame` calls.

If we profile the strategy on the left by initiating the recording, clicking the
"Execute" button, and then stopping the recording, we will see something like
the above image. Notice that before the green paint bar, there are two `Layout`
bars. Because we perform a write in JS and then query the element's
layout/position, we trigger a forced synchronous layout. This is what is
responsible for the first layout bar. The second one is caused by the final
write which invalidates the previous one and causes the browser to perform a
recalculation of the layout before the frame is produced.

Performing a profile of the strategy on the right hand side reveals a subtle
difference. Instead of seeing two layout bars before the relevant paint occurs,
only one occurs. In this example, the forced synchronous layout found in the
previous example is eliminated by batching the reads and writes together and
performing the reads before the writes. The browser will still need to calculate
the layout, but it only needs to do this once before the frame is produced
because there aren't any reads that ask it to calculate the layout earlier than
that. As a result, this is a more efficient way to render a frame than the
previous example.

## Layout Thrashing

When the browser performs multiple forced synchronous layouts in the same frame,
the effects on performance become particularly noticeable. This effect, known as
"layout thrashing", commonly occurs when a forced synchronous layout (a write
followed by a read) is placed inside a loop.

```js
const boxes = document.querySelectorAll(".box");

boxes.forEach((box) => {
  // Write
  box.classList.add("box--large");
  // Read
  const width = box.getBoundingClientRect().width;
});
```

<div>
  <a href={demoUrl}>Layout Thrashing Demo</a>
</div>

Let's go back to the same demo page as before and profile what layout thrashing
looks like by following these steps:

1. Click the "Add 1,000 cloned nodes" button to add 1,000 more boxes to the
   page.
2. Open the DevTools performance panel.
3. Click the "Record" button on the performance panel to start profiling.
4. Click the "Execute" button on the left hand side.
5. Click the "Stop" button on the performance panel to stop profiling.

In the profile produced, you'll noticed that over a thousand layout calculations
are made before paint occurs! Perhaps more importantly, these calculations cause
the click handler to be considerably slow. In my profile with a 6x CPU throttle,
it takes around 10 seconds for the click handler to complete! This blocks the
main thread and prevents the user from performing other actions on the page
(e.g. clicking, highlighting text, etc).

Now lets profile the strategy on the right hand side to compare. Refresh the
page and perform the same steps as above but, as part of step 4, click the
"Execute" button on the right hand side instead of the left hand side.

You'll notice that only one layout bar appears before the paint in this profile.
Additionally, the time it takes to reach the paint bar is a lot faster. In my
profile with a 6x CPU throttle, it is around 50x faster than the previous
profile and only takes around 200ms.

As this example demonstrates, the treatment for layout thrashing is to avoid any
forced synchronous layouts. Batch the reads together, batch the writes together
and defer the execution of the writes to another frame. That's how we eliminate
the layout thrashing and make substantial improvements to the performance.

## Conclusion

Now that you have learned how to recognize inefficient animations, forced
synchronous layouts, and layout thrashing through performance profiles, the next
step is to use this knowledge in the wild. Profile websites that are in
production and see if you can spot any of these inefficiencies. Better yet, fix
any issues you observe and improve the website's user experience for all. Have
fun!
